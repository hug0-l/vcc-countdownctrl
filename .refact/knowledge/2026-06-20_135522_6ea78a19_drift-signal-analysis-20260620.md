## Drift Signal Analysis — 2026-06-20

**Changed files:** 19 (all in `.refact/`)
**Additions:** 29,730  |  **Deletions:** 61
**Application code changed:** 0 files

### Subsystem breakdown (path groups):
- `.refact/tasks/` — trajectory agent logs (largest contributor, e.g. agent-T-2-13411e43.json alone = +4,844 lines)
- `.refact/buddy/chats/` — conversation JSON files (17+ workflow agents: error_detective, docs_gardener, setup_coach, etc.)
- `.refact/buddy/memory_ops.jsonl` — growing memory operation log (565+ ops)
- `.refact/buddy/runtime_queue.jsonl` — runtime event queue
- `.refact/buddy/diagnostics.jsonl` — error diagnostics (renderer crash breadcrumbs)
- `.refact/knowledge/` — knowledge markdown files (file analysis reports, error investigation memos)

### Cross-cutting risk:
- **Unbounded growth**: Trajectory agent logs and conversation files accumulate without retention policy. Single agent file can be +4.8K lines.
- **Noisy "drift" signal**: 100% of churn is tool metadata — the actual project (server.py, templates/index.html) has zero changes. This dilutes real architecture drift detection.
- **Low deletion ratio**: 61 deletions vs 29,730 additions (0.2%) means no cleanup/compaction logic.

### Suggested guardrails:
1. Add `.refact/tasks/*/trajectories/` and `.refact/buddy/chats/` to a gitignore or drift-watch exclusion list for application-level architecture monitoring.
2. Consider a retention/compaction policy for trajectory logs (e.g. keep last N, or summarize old ones).
3. Split the drift watcher signal: track application code drift separately from tool metadata churn.
